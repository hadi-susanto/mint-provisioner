# Requirements

### Overview & Goals
Rename module category directories, add category-level `metadata.conf`, and update module listing output to use category metadata labels and fixed table widths.

### In Scope
- Rename directories under `modules/`:
  - `desktop-utilities` → `desktop`
  - `software-engineering` → `development`
  - `terminal-experience` → `terminal`
  - `system-administration` → `sys-adm`
- Add `metadata.conf` in each category directory (`modules/<category>/metadata.conf`) with:
  - `NAME` = human-readable long form from previous category name
  - `DESCRIPTION` = short category description
- Update `lib/metadata_parser.sh:list_available_modules()` output format to:
  - `Module Category: <NAME from category metadata>`
  - `No | Name | Source | Status | Description`
  - right-aligned `No`
  - column widths: `3, 20, 12, 12, *`
- Keep module listing behavior that still emits canonical IDs to stdout for script consumption.
- Update docs affected by category rename (at minimum `README.md` examples and `modules/README.md` category references), following project guidelines.

### Out of Scope
- Changing per-module lifecycle scripts beyond path adjustments caused by directory rename.
- Changing module-level metadata schema (`NAME`, `DESCRIPTION`, `SOURCE`).

# Technical Design

### Current Implementation
- `lib/metadata_parser.sh`:
  - `list_module_directories()` discovers modules at depth 2 (`modules/<category>/<module>`).
  - `list_available_modules()` currently prints section header from folder slug (`$category_name`) and table columns using widths `3, 32, 12, 12, *`.
  - `parse_config()` reads only module `metadata.conf` (`$MODULES_DIR/$canonical_module_id/metadata.conf`).
- Existing category docs still use old category names in `modules/README.md` and category markdown files (`modules/DESKTOP-UTILITIES.md`, etc.).
- Entrypoints `install.sh` and `configure.sh` rely on canonical ids generated by `list_available_modules()` and resolver logic in `metadata_parser.sh`.

### Key Decisions
- Introduce category metadata as separate files at `modules/<category>/metadata.conf` (not mixed into module metadata).
- Extend listing logic in `lib/metadata_parser.sh` to parse category metadata for display label/description while preserving canonical ID emission.
- Preserve existing module metadata parsing contract and resolver behavior; only adapt paths indirectly through renamed category folders.

### Proposed Changes
1. **Category directory migration**
   - Rename folder paths and keep module subfolders intact:
     - `modules/desktop-utilities/*` → `modules/desktop/*`
     - `modules/software-engineering/*` → `modules/development/*`
     - `modules/terminal-experience/*` → `modules/terminal/*`
     - `modules/system-administration/*` → `modules/sys-adm/*`
2. **Category metadata files**
   - Create:
     - `modules/desktop/metadata.conf` (`NAME="Desktop Utilities"`)
     - `modules/development/metadata.conf` (`NAME="Software Engineering"`)
     - `modules/terminal/metadata.conf` (`NAME="Terminal Experience"`)
     - `modules/sys-adm/metadata.conf` (`NAME="System Administration"`)
   - Add concise category `DESCRIPTION` in each file.
3. **Listing output rewrite (`lib/metadata_parser.sh`)**
   - Add category metadata reader helper for `modules/<category>/metadata.conf`.
   - Change section header to `Module Category: <category NAME>`.
   - Replace current table header with exact requested row:
     - `No | Name | Source | Status | Description`
   - Enforce formatting widths in data rows:
     - `%3s %-20s %-12s %-12s %s`
   - Keep `No` right-aligned and reset numbering per category section.
   - Keep canonical id `echo "$canonical_id"` behavior untouched for callers.
4. **Documentation alignment**
   - Update category references and examples in:
     - `README.md` (usage examples)
     - `modules/README.md` (category list/links)
   - Update category markdown filenames/links if needed to avoid stale old-name references.

### Risks & Mitigations
- **Risk:** Category-level `metadata.conf` could be accidentally treated as a module.
  - **Mitigation:** Keep module discovery at depth 2 and parse category metadata only from direct category path.
- **Risk:** Broken docs/links after rename.
  - **Mitigation:** Update both root and module docs in the same change set.

# Testing

### Validation Approach
Run script-level and behavior checks focused on listing format and selector compatibility after renaming categories.

### Key Scenarios
- `./install.sh` with no args shows category sections titled by category `NAME` and table columns in requested format/widths.
- `./configure.sh` with no args still consumes canonical IDs correctly from `list_available_modules()` output stream.
- Canonical selectors using new category slugs (e.g., `development/git`, `terminal/eza`) resolve correctly.

### Edge Cases
- Missing/invalid `modules/<category>/metadata.conf` is handled with clear logging/fallback behavior in listing.
- Category numbering resets from `1` for each category section.

### Checks
- `bash -n` on modified scripts.
- Smoke run of listing and selector resolution paths.
